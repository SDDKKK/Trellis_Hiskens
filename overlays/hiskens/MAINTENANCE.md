# Hiskens Overlay Maintenance Guide

## Core Principles

1. Never edit upstream base templates in place just to carry Hiskens behavior.
2. Keep custom behavior inside `overlays/hiskens/`.
3. Treat upstream sync as a review workflow, not an auto-merge workflow.
4. Re-run sandbox validation before any production cutover.

## Handling an Upstream Sync PR

1. Read the PR body generated by `scripts/upstream-diff.sh`.
2. Check whether upstream changed hooks, CLI commands, configurators, or `packages/cli/src/utils/overlay.ts`.
3. Confirm `.upstream-version` matches the tag named in the PR title.
4. If merge conflicts appear, resolve them on the sync branch only. Never patch `main` directly.
5. After review, test the synced branch locally before merging the PR.

## OVERRIDE File Adaptation Flow

The following upstream paths require manual comparison when they change:

- `packages/cli/src/templates/claude/hooks/*.py`
- `packages/cli/src/commands/*`
- `packages/cli/src/configurators/*`
- `packages/cli/src/utils/overlay.ts`

Adaptation steps:

1. Diff upstream file against the current fork branch.
2. Diff the matching overlay file under `overlays/hiskens/templates/...`.
3. Copy only the upstream behavioral changes that are still compatible with Hiskens customizations.
4. Keep Hiskens-only logic isolated and documented in commit message or PR notes.
5. Re-run targeted tests plus the Phase 4 sandbox flow after adapting overrides.

## New Upstream File Classification

When upstream adds a file, decide which bucket it belongs to:

- `BASELINE`: keep upstream file unchanged and let the base template own it.
- `APPEND`: keep upstream base file, then layer Hiskens additions through overlay templates or settings merge.
- `EXCLUDE`: do not install the upstream file in Hiskens projects because a project-local or deprecated replacement already exists.

Decision checklist:

1. Is the file generic and safe for all Hiskens consumers? Choose `BASELINE`.
2. Does the file need Hiskens-specific behavior on top of upstream? Choose `APPEND`.
3. Would the file reintroduce deprecated commands or conflict with project-local assets? Choose `EXCLUDE`.

If `EXCLUDE` is chosen, update `overlays/hiskens/exclude.yaml` and explain why in the sync PR.

## Updating Overlay Content

1. Change only files under `overlays/hiskens/`.
2. If a new platform or template needs overlay content, mirror upstream directory structure exactly under `overlays/hiskens/templates/`.
3. For settings changes, prefer `settings_merge` targets defined in `overlays/hiskens/overlay.yaml`.
4. Update this guide if the review flow or classification rules change.

## Testing Overlay Changes

Use the Phase 4 sandbox process before merging overlay maintenance work:

1. Create a temporary sandbox with `trellis init --overlay hiskens`.
2. Run the P0 and P1 validation matrix.
3. Run `trellis update --overlay hiskens` against a modified overlay-managed file.
4. Run the rollback drill and confirm `.trellis/scripts/get_context.py` still works.

Reference: consumer-project validation report at `.trellis/tasks/04-13-trellis-sandbox-validation/validation-report.md`.
